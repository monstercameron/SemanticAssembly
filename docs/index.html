The premise

It's not that the model can't read assembly.

Raw assembly is hard for an LLM to edit reliably because correctness lives in long-range, implicit, stateful dependencies — exactly the regime where attention degrades:

Which value is in a1 here?

It depends on every prior instruction. Reconstructing it means attending across the whole function.

What does this call clobber?

An ABI contract that lives only in the programmer's head — and a value left in the wrong register is silently destroyed.

Is the frame still 16-byte aligned?

Stack offsets are magic numbers that couple instructions invisibly.

Did this branch keep the CFG intact?

Control flow is implicit in label position and fall-through.

The thesis. Semantic Assembly is an agent-native machine format: machine code an agent can understand (every load-bearing fact is written down, locally, instead of left implicit in machine state), edit (every fact is an addressable row with a stable handle, and a wrong edit produces a targeted diagnostic instead of silent corruption), and manage (handles survive edits, emission is deterministic, every diagnostic has a stable code). Assembly has none of the hand-holding of C — so the safety C buys by thickening its abstraction is recovered here from metadata plus a validator, with the layer kept thin. The price is tokens, and the price is accepted; what is never accepted is a fact the model might wrongly believe.
The idea

Same instruction. Every assumption written down.

It is thin: each instruction row still maps to exactly one real assembler statement. But the opaque parts become facts.

raw RISC-V
add a0, a0, a1
ret
semantic assembly (.sasm)
left is value
left type Int64
right is value
right type Int64
result is value
result type Int64

Add2 is function
Add2 symbol add2
Add2 in left  Int64 a0  # arg arrives in a0
Add2 in right Int64 a1
Add2 out result Int64 a0
Add2 effect none       # asserted: no memory/calls

addLeftAndRight is insn
addLeftAndRight operation Add
addLeftAndRight destination a0
addLeftAndRight firstSource a0
addLeftAndRight secondSource a1
addLeftAndRight reads left    # a0 holds `left` here
addLeftAndRight reads right
addLeftAndRight writes result

returnResult is insn
returnResult operation Return
returnResult returns result

Names are full words by design — operation/destination/firstSource (never op/rd/rs1), descriptive instruction handles (never I1). The audience is an agent reading and a validator checking.

The projection

Verbosity is free at runtime.

Lowering is a function π : .sasm → .s that discards all the context, keeping only the minimal facts that form real instructions/operands/labels/data. The verbosity lives above the projection line and never reaches the assembler — it costs source bytes, not code.

Two tiers of fact

  • Authoritative — survives π: the operands, opcode, labels, data.
  • Asserted (S-check) — stripped, but the validator verifies it: effect none, liveOut, types.
  • Intent (S-intent) — the irreducible why: purpose, meaning. The only facts allowed to go unchecked.

Invariant: strip every asserted/intent fact and the emitted .s is byte-identical.

π(add2.sasm) → exactly this
	.text
	.globl	add2
add2:
	add	a0, a0, a1
	ret

Deterministic and byte-identical to the hand-written golden — so an agent's edit is a minimal, reviewable .s diff.

The discipline

Surface dangerous state — not everything.

More facts is not the goal, and token count is not the constraint — trust is. A fact that merely restates what's derivable from the op tables is an unchecked copy that can drift stale, and is forbidden. The rule: every stored fact is consumed by the emitter, checked by the validator, or marked as intent — no fact is silently believed.

A fact earns its place iff it (1) affects correctness across more than one instruction, (2) crosses a control-flow edge, (3) crosses a call / syscall / ABI boundary, (4) names a memory region / stack slot / aliasing assumption, (5) is required to prove a contract (effect, leaf, stack, return, preserve), or (6) is irreducible intent. Otherwise: derive it, don't store it.

The sharpest case

reads a1 is forbidden — a register use is already in the op table (derivable). reads left is kept — it asserts the register↔value binding at this point, the long-range fact raw assembly hides. Value names yes; register names no.

The payoff

The validator is the point.

Syntax isn't the value — checking is. The validator recomputes the truth from authoritative facts (a CFG, forward and backward liveness, a value-flow may-analysis) and judges every assertion against it. The full diagnostic catalog — 19 stable codes, each naming the entity handle at fault:

E-ISA-OPCODEE-ISA-REGE-ISA-FIELDE-REF E-IMM-RANGEE-ORDER-MIXEDE-ABI-ALIGN E-ABI-PRESERVEE-LEAFE-EFFECTE-TYPE W-SLOTE-CFG-EDGEE-LIVE-UNDEFE-LIVE-RET W-DEADW-CLOBBERE-VALUE-FLOWE-DERIVABLE

The bug assembly hides best

Recursive fib deliberately moves number into a callee-saved register so it survives two calls. Leave it in caller-saved a0 — and the value-flow analysis catches the silent clobber:

$ python -m sasm check naive_fib.sasm
error E-VALUE-FLOW computeNumberMinusTwo: reads number but ['a0']
      hold ['·call:callFibNumberMinusOne:a0']

A value destroyed across a call — normally invisible — surfaced mechanically, pointing at the exact instruction. That is the loop the format exists for: edit → validate → diagnostic (with a handle) → re-edit.

Honest status

What works, and what doesn't yet.

Working today

  • Compiler for RV64 "Tier A" (RV64I + M + Zicond + scalar atomics) — byte-identical, assembling, executing.
  • Validator: the entire 19-code catalog, zero false positives on the examples.
  • Self-contained Docker eval environment.
  • Pure-Python toolchain, no dependencies.

Not done yet

  • General E-DERIVABLE reachability linter (research-grade).
  • Broader ISA coverage (Tiers B/V/P) via the official riscv-opcodes generator.
  • Compiler edges the examples don't exercise: multi-function files, pipe sugar, ordinal ordering.
It's a bet, stated as one. The claim — that explicit, addressable, checked facts beat inferred ones — is falsifiable. The planned benchmark runs four arms carrying the same information in different forms: raw .s, .s + a prose fact block, .s + the same facts as inline comments, and .sasm — across comprehension, long-range edit, and stale-fact tasks, both one-shot and with a validator-in-the-loop at equal feedback budgets. The critical comparison is .sasm vs inline comments: it isolates addressability + checkability from mere locality. If .sasm doesn't beat good comments once the loop closes, the format is overbuilt — a result worth knowing.
